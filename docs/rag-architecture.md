# 知识库 RAG 架构

## 用户界面的证据边界

检索阶段的候选切片并不是用户可参考的来源。后端只在生成完成后，按答案中的合法 `[S#]` 引用筛选 `done.sources`；前端的证据检查器和引用抽屉只渲染这些已采纳切片。候选数量和阶段耗时仍可在“检索过程”中查看，但不会以文档卡片形式干扰用户判断。

文档上传采用异步索引任务。原文件写入成功后立即创建 `pending` 任务，依次经历“解析文档、拆分证据、写入知识库”，最后写入 `done`、实际切片数和完成时间。知识库管理页每 1.5 秒轮询未完成任务，因此用户不用刷新页面也能看到入库成功。

## 数据流

```text
上传原文件 → 解析 → 分块 → pgvector
用户问题 → QueryPlan（显式过滤 + 按需改写）→ 向量/关键词候选
         → 主问题/变体加权 RRF → 去重与单文档配额
         → 证据截断 → 带 [S#] 引用生成 → 引用与耗时回传
```

## QueryPlan 与受控召回

`QueryPlan` 先用规则从原问题提取明确的 `filename`、页码范围、Sheet 与文件类型；这些条件以参数化 SQL 下推到 pgvector，LLM 无权新增或修改它们。时间过滤暂不支持，因为当前索引尚未保存可检索的文档日期元数据。

普通独立问题只用原问题检索。只有明显追问，或首次过滤后少于 3 个候选时，才使用会话记忆生成一个标准问题和至多 2 个同义变体。主问题权重高于变体；多路结果经过加权 RRF、`chunk_id` 去重和单文档最多 2 个候选限制。规划、过滤或变体检索失败时，保留原问题与原有显式过滤条件回退，绝不为“找到结果”扩大检索范围。

每次查询会写入内部结构化 Trace（规划来源、过滤摘要、查询数、候选数、耗时和回退原因），但候选与规划正文不会进入 SSE 或用户界面。

当前 34 条金标集的 32 条可检索样本上，QueryPlan 与旧 hybrid 的文档级 Recall@5、MRR、NDCG@5 均为 1.000；平均检索延迟为 303.0ms，对比旧 hybrid 的 580.4ms。该结果仅覆盖当前 3 份已索引文档，应在知识库扩充后重新评测。

## 三层记忆

记忆只帮助系统理解追问和用户期望，不是知识库证据，不能支撑回答中的任何事实性结论。

1. 短期记忆：`conversations` 保存当前 `user_id + session_id` 的已完成问答；查询时只读取最近 6 条，避免提示词无限增长。
2. 长期记忆：对话超过 8 条时，从最近 16 条问答生成抽取式滚动摘要并保存到 `conversation_summaries`。摘要不调用模型，不产生额外幻觉；它只参与追问改写。
3. 用户记忆：用户通过 API 明确保存的表达偏好，例如“回答使用中文”。偏好保存在 `user_facts`，可以查询和删除；匿名用户不读写该类记忆。

持久化真相源是 MySQL：`conversations`、`conversation_summaries`、`user_facts`。PostgreSQL + pgvector 仍只保存知识库切片与检索索引。Redis 是可选缓存：命中时加速读取，未命中从 MySQL 回填；Redis 或 MySQL 任一不可用时，RAG 主链路继续以进程内缓存运行。

## 核心原则

1. PostgreSQL + pgvector 是唯一的知识库真相源；不再回退到 ChromaDB，不再执行 GraphRAG、Agentic RAG 或办公 Agent 路由。
2. 每个切片拥有 `chunk_id`，每个原文件拥有稳定的 `document_id`。来源不按文件名合并，因此同一 PDF 的不同页可以分别引用。
3. 向量候选和关键词候选各取前 30 条，再做一次 Reciprocal Rank Fusion。显式文件名、页码、Sheet、文件类型分别使用受控 SQL 条件与索引；`vector_documents.search_vector` 有 GIN 索引，必要时还有 trigram 索引辅助中文子串召回。
4. 只有明显追问或弱信号检索才使用最多四轮简短历史生成 QueryPlan；普通问题绝不额外调用 LLM，且记忆不能改变用户显式过滤条件。
5. 上下文按证据块截断，每块最多 1200 字符。模型只能使用 `[S1]` 等实际存在的编号；无证据或无效引用时返回固定的资料不足回答。
6. 会话摘要和用户偏好只被传入追问改写上下文，绝不直接拼入知识库回答提示词。

## 性能策略

- 启动后台预热 BGE embedding。
- 默认 `RAG_ONLINE_RERANK=false`。本地 CPU Cross-Encoder 只能用于离线评测，不进入线上问答。
- SSE 先发送 `retrieval` 事件（仅候选数量、查询数与阶段耗时），再输出 token，最后发送仅含已采纳引用的 `done` 事件。

## 反馈闭环

`POST /api/rag/feedback` 写入 `data/rag-feedback.jsonl`。每条反馈关联问题、回答、来源、引用编号和判定结果，可回放为评测样本。
