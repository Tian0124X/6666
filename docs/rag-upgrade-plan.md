# RAG 生产化升级方案

## 目标

将当前“单次检索 + 生成”的 Demo 链路，升级为可持续优化、可追溯、可评测的知识库问答系统。所有回答仍以最终采纳的引用为唯一可见证据。

## 当前基线与主要缺口

当前链路已经具备异步文件解析、中文分块、向量与关键词 RRF 融合、QueryPlan 受控多路召回、ContextBuilder、可信引用闸门、脱敏 Trace、带 SLA 看板、负责人分派与人工结案的 JSONL 反馈队列，以及三层会话/用户记忆。文档可在上传时标注人工确认的生效/发布日期，日期与版本哈希会随切片持久化，QueryPlan 可据此安全下推时间过滤。当前剩余工作属于持续运营：为历史文档补齐人工日期、随新增真实资料扩大金标覆盖面、积累当前运行批次 Trace，并根据真实负载校准 P95 与 SLA 告警阈值。

## 按模块的优化方案

下表逐项对应需求图中的模块。`现状`基于当前代码，`首个改动`是建议实际落地的最小、可验证版本。

| 模块 | 当前现状 | 关键缺口 | 首个改动 | 验收指标 |
| --- | --- | --- | --- | --- |
| 知识工程 | 已支持 PDF、Word、Excel、TXT 解析、类型化分块、质量报告、版本哈希和人工文档日期；上传后异步写入 pgvector | 历史文档尚无人工日期，新增资料仍需人工抽检解析质量 | 上传时记录“页数/字符数/切片数/空页数/解析器/版本哈希/人工日期”，重传按源文件覆盖旧切片 | 抽检文档的标题、表格、页码不丢失；空切片为 0；同一文档重传不产生重复切片 |
| Query 改写 | 已实现 `QueryPlan`：规则提取文件、页码、Sheet、文件类型和明确日期；追问或弱信号时生成标准问题和最多 2 条变体 | 尚无意图分类；时间过滤只覆盖人工确认日期 | 继续用真实金标集验证改写收益，并在新增资料入库时补齐日期 | 普通独立问题不调用模型；失败回退原问题且不放宽过滤 |
| 检索召回 | pgvector 向量/关键词 RRF、参数化元数据过滤、跨查询加权 RRF、`chunk_id` 去重和单文档配额已实现 | 需要在更大语料上复验召回 | ContextBuilder 已处理相邻块拼接与预算分配 | 固定评测集 Recall@5、MRR、NDCG 不低于基线；正确文档不被重复块挤出前 5 |
| Rerank | 已接入本地 `BAAI/bge-reranker-v2-m3`、批量打分、超时保护和 RRF 回退；线上默认关闭 | 当前小型金标集未证明排序收益 | 新文档进入后继续扩充难例金标集，再做离线 A/B | 仅在 NDCG@5 提升且延迟满足预算时开启；否则保持关闭 |
| 截断策略 | 已实现 `ContextBuilder`：全局 token 预算、得分优先、切片去重、同文档配额与相邻块拼接 | 还需在更大语料与长文档上观察预算配置 | 用真实问答与后续 Trace 验证每条入模证据 | 提示词不超预算；每条入模证据都有可回溯引用；关键证据不会因前置长文本被截掉 |
| 可信生成 | 已实现证据最小充分性、逐实质性结论的引用覆盖校验、单次受控重生成及答案级金标评测 | 需要随新资料持续抽检和扩展金标 | 持续评测拒答正确率、引用覆盖率和结论支持率 | 无证据问题拒答率达标；引用合法率 100%；人工抽检的结论-证据支持率持续提升 |
| 过程评测 | 已回传阶段耗时，并有 Recall 和 citation accuracy 的基础工具 | 未记录每次查询的改写、候选排名、重排、截断和最终引用，无法定位错误环节 | 增加 `trace_id` 与结构化 Trace：每阶段输入、输出摘要、耗时、版本；建立覆盖制度、表格、追问、拒答的金标集 | 任一差评可归因为知识、改写、召回、重排、截断或生成；每次发布自动跑评测集 |
| 反馈闭环 | 差评自动归因并进入待办；管理员可分派负责人、按“金标/知识工程/检索调优/忽略”人工结案，结论和分派均写入审计日志 | 仍需在真实运行数据上确定告警阈值与负责人轮值 | 看板显示 SLA 时限、逾期数和未分派数；人工确认后再创建金标或知识工程任务 | 差评有处理状态、归因和负责人；同类问题的下一轮评测提升；禁止未经审核自动写入知识库 |

## 记忆在 RAG 中的正确位置

会话记忆解决的是“用户说这个、它、继续时到底指什么”，不能代替知识库事实，也不能作为回答证据。短期会话、抽取式长期摘要和显式用户偏好均已完成，并只作为 `QueryPlan` 的追问消歧输入。

长期记忆应当最后建设，并拆为两类：

1. 会话摘要：只压缩已完成会话的目标、已确认结论和未解决问题，用于长对话恢复。
2. 用户偏好：仅保存用户明确表达且可撤回的偏好，例如“回答使用中文、优先给制度原文”。

两类记忆均不参与知识事实引用，不自动从模型猜测中写入。

当前实现进一步明确为三层：短期会话保存最近消息、长期会话以抽取式摘要压缩、用户记忆只保存显式提交且可删除的表达偏好。持久化使用 MySQL，Redis 仅作为可选读取缓存；PostgreSQL + pgvector 保持为知识库检索存储。

## 推荐实施顺序

### 第 0 步：先建立基线，防止盲目调参

从现有文档中人工整理至少 30 条金标问题，覆盖：制度问答、表格字段、跨页信息、同义问法、追问、知识库无答案。每条标注正确文件、页码或切片，以及应拒答与否。先跑出当前的 Recall@5、MRR、NDCG@5、引用合法率、拒答正确率、P95 总延迟。

已提供 `data/eval/rag-golden.jsonl`：基于当前 PDF、Excel、Word 文档人工核验的 34 条样本，覆盖制度、表格字段、页码、Sheet、追问与拒答。可运行：`python -m app.eval.retrieval_eval --strategy hybrid --dataset data/eval/rag-golden.jsonl`。模板仍保留在 `rag-golden.template.jsonl`，供后续扩充。

### 当前真实检索基线（2026-07-15）

本机 pgvector 已索引 3 份文档、108 个切片（Excel 101、PDF 6、Word 1）。金标集中 32 条带有相关文档的样本用于文档级检索指标；2 条拒答样本保留给后续可信生成评测。评测已按文件名去重，避免同一文档的多个切片虚高指标。

| 策略 | Recall@5 | MRR | NDCG@5 | Hit Rate@5 | 平均检索延迟 |
| --- | --- | --- | --- | --- | --- |
| hybrid（旧链路） | 1.000 | 1.000 | 1.000 | 1.000 | 580.4ms |
| query_plan（新链路） | 1.000 | 1.000 | 1.000 | 1.000 | 303.0ms |

### 重索引后线上复验（2026-07-16）

新版服务完成 3 份既有文档重索引后，文档列表已持久化展示每份文档的 SHA-256 与索引时间，仓库占位文件 `.gitkeep` 不再暴露。真实 `query_plan` 检索评测的 32 条可检索样本仍为 Recall@5 1.000、MRR 1.000、NDCG@5 1.000、Hit Rate@5 1.000，平均检索延迟 1110ms。该数字包含首次 BGE 模型冷启动以及部分追问/弱信号问题的受控 LLM 改写，不能直接同旧的热态 303.0ms 做等价比较；线上热态延迟继续以当前 `runtime_id` 的 Trace P50/P95 观察。

结论：当前小型金标集上，QueryPlan 不降低文档级召回和排序指标，平均检索延迟降低约 48%。后续新增文档后必须扩充金标集，不能把该结果外推为通用结论。

### 第 1 步：补齐知识工程质量报告

修正分块策略实际未按文件类型生效的问题，并把上传结果从“成功/失败”升级为可检查的索引质量报告。这一步不改模型，优先解决“知识有没有被正确放进去”。

本次实现默认按 PDF、MinerU PDF、Word、Excel 和通用文本使用不同预设；显式传入 `chunk_size` 与 `chunk_overlap` 的旧调用仍保持原行为。上传完成状态会新增 `quality`，包含文件 SHA-256、解析单元、页数、源文本字符数、切片数、空切片、平均切片长度、解析器与覆盖的文件类型。文件 SHA-256 与索引时间会持久化到每个切片元数据，`GET /documents` 可在服务重启后返回 `version`，便于确认同文件名重传是否已覆盖旧版本。上传人可选填 ISO 格式的“文档生效/发布日期”；只有该人工确认字段会写入 `document_date` 并允许 QueryPlan 做日期过滤，文件名、文件系统时间、解析器和模型都不会猜测或补造日期。

### 第 2 步：实现 QueryPlan 与受控多路召回

已完成。规则层提取文件名、页码、Sheet、文件类型和用户明确给出的年月/年月日，并以下推参数过滤；日期条件只匹配上传时人工确认的 `document_date`。只有追问或首次候选不足 3 条时调用 LLM。主问题与至多 2 条变体并行检索并做加权 RRF，最终按切片去重并限制单文档最多 2 条候选。规划、过滤或变体失败均回退到原问题且不移除显式过滤。过程仅写入内部 Trace，不扩展用户可见 SSE。

### 第 3 步：先离线验证 Rerank，再决定是否上线

已完成真实 Cross-Encoder 接入与离线验证。使用本地缓存的 `BAAI/bge-reranker-v2-m3` 对 QueryPlan 候选批量打分；线上路径提供候选阈值、超时和 RRF 回退，但默认关闭。

当前 32 条可检索金标样本中，`query_plan` 与 `query_plan_rerank` 的 Recall@5、MRR、NDCG@5 均为 1.000；Rerank 平均延迟为 1434.1ms，高于基线的 800.9ms。因此当前不满足“有收益且满足延迟预算”的上线条件，`RAG_ONLINE_RERANK` 保持 `false`。此结论只适用于当前 3 份文档的小型集；新增高混淆度文档后必须重新评测。

### 第 4 步：引入 ContextBuilder 与可信生成闸门

已实现。`ContextBuilder` 按全局 token 预算选择证据，优先使用 Rerank/RRF 分数，执行切片去重、同文档配额，并将相邻切片同源拼接；只有真正进入上下文的切片才会进入可引用来源。

生成前要求证据达到最小字符阈值，且命中至少一个有效问题词；无关召回会固定返回“资料不足，无法依据当前知识库确认。”且不返回来源。生成后校验每个实质性结论句都含有输入来源中的合法引用。校验失败时最多重生成一次，仍不合格则只降级展示可核查来源、不输出未经验证的事实结论；会话记忆绝不补写知识事实。答案级金标评测和人工抽检将在第 5 步接入。

### 第 5 步：Trace、评测与反馈闭环

已实现脱敏 JSONL Trace：保存 `trace_id`、问题指纹、QueryPlan 摘要、候选数、最终采纳引用、结果模式与阶段耗时；不保存原始问题、规划正文或未采纳候选，也不扩展 SSE。`POST /api/rag/feedback` 会保留全部反馈，并将差评按知识缺口、回答质量或引用错误自动写入待人工处理队列。管理员可通过前端 `/feedback` 页面或 `GET /api/rag/feedback/queue` 查看队列、SLA 时限、逾期数和未分派数，可用 `PATCH /api/rag/feedback/{feedback_id}/assignment` 指定负责人，并通过 `POST /api/rag/feedback/{feedback_id}/resolve` 以“金标集、知识工程、检索调优或忽略”结案；分派与结论均追加到 `data/rag-feedback-actions.jsonl`，不会自动更新金标、索引或知识库。SLA 时限由 `RAG_FEEDBACK_SLA_HOURS` 配置，默认 72 小时。

答案级金标现已补充关键事实词、允许引用文档和拒答契约，并新增确定性评测器。2026-07-16 在完成 3 份文档、108 个切片重索引后，34 条样本的单进程真实评测达到：拒答正确率 1.000、引用覆盖率 1.000、引用合法率 1.000、关键事实词覆盖率 1.000、结论-证据支持率 1.000；平均端到端延迟 1688.7ms、P50 1122.6ms、P95 2810.7ms。已修复版本号小数点被误切句、预测请求被历史数据误答、Excel 精确 ID 召回、MinerU 表格行缺失和跨分块条款证据缺失；相邻证据使用 QueryPlan 的受控变体排序，且不再叠加父切片的词项分。评教首步样本已人工复核为“进入新版教务系统并登录”，不再要求其后的评价入口词；年假审批顺序金标只要求审批人链，不将 OA 提交入口误算为审批步骤。

单进程评测命令为：`python -m app.eval.answer_eval --progress --output data/eval/rag-answer-eval.json`。命令逐条输出脱敏金标 ID，并在报告中记录平均、P50、P95 延迟；该 JSON 是运行产物，不应提交到代码仓库。桌面工具的单次命令等待上限为 60 秒时，应在用户终端持续运行该命令，避免将分批冷启动延迟误作线上性能基线。

线上运行态则通过 `/api/rag/diagnostics` 的 `recent_latency` 观察当前应用运行批次最近 100 条脱敏 Trace 的检索、生成和总耗时 P50/P95。该数据以 `runtime_id` 隔离重启前后的历史样本，用于告警与趋势判断；当当前批次累计至少 20 条样本后，`latency_alert` 会以默认 3000ms 总耗时 P95 预警线标记 `normal` 或 `alert`，样本不足时明确返回 `insufficient_samples`。发布门槛仍以单进程金标评测报告为准。

## 分阶段路线

| 阶段 | 能力 | 解决的问题 | 验收标准 |
| --- | --- | --- | --- |
| 1（已完成） | 持久化会话记忆 | 追问无法理解、刷新后上下文丢失 | 相同 `session_id` 的后一问会自动带入近期对话；不同用户和会话隔离 |
| 2（已完成） | QueryPlan 与受控多路召回 | 用户表达与文档措辞不一致、显式条件未生效、重复块挤占 | 改写只在明确追问或候选不足时触发；过滤、候选数、回退原因写入内部 Trace |
| 3（已完成） | 分层召回与重排 | 关键证据排不进前列 | 向量、关键词、元数据过滤融合后，以离线评测集验证 Recall@5 和 MRR；Rerank 未达收益门槛，保持线上关闭 |
| 4（已完成） | 可信生成 | 检索到了仍可能回答失真 | 每个结论可回溯至引用；无充分证据时拒答；引用覆盖率达标 |
| 5（已完成） | 过程评测与反馈闭环 | 不知道错误发生在哪一环 | 固定评测集覆盖检索、引用、回答和延迟；差评可回放、归因并由管理员人工结案 |
| 6（已完成） | 三层记忆 | 对话很长、用户偏好无法复用 | 会话摘要与明确用户偏好分开存储；MySQL 持久化、Redis 可选缓存 |

## 阶段 1：会话记忆边界

- 使用前端生成的稳定 `session_id` 作为会话边界；点击“新建问答”才创建新会话。
- 服务端以 `user_id + session_id` 作为读取条件，最多取最近 6 条已完成消息，防止上下文无限膨胀和串会话。
- 用户问题在检索前写入；助手回答只在流式生成正常结束后写入。
- MySQL 可用时持久化到 `conversations`；不可用时退化为进程内缓存，主问答链路不中断。
- 当前已在阶段 1 基础上增加抽取式会话摘要和显式可删除的用户偏好；两者都不作为知识库事实或引用。

## 后续关键设计原则

1. 检索记忆与用户记忆分离：会话上下文只辅助理解问题，不能替代知识库证据。
2. 只在必要时调用大模型改写和重排：普通独立问题走低延迟路径。
3. 每一次回答都记录检索候选数、改写、重排、引用校验和总耗时，作为评测与反馈归因依据。
4. 用户反馈优先进入离线评测集和问题队列，不直接自动修改知识库或长期记忆。
